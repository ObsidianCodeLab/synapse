# 研发会议室智能体交互走查记录

## 1. 目标与范围

本文用于梳理 `rd_meeting`（研发会议室）相关代码中，围绕“智能体交互”方向存在的不合理设计、潜在风险，以及建议的改造方案，便于后续逐项核对。

本次走查重点关注以下链路：

- 会议室主控 Host 与 Worker 的协作与委派
- 智能体会话、上下文、角色识别
- 计划（work plan）与节点执行（node run）的交互门禁
- 智能体之间的消息传递能力是否真实可用
- 工具裁剪、执行隔离与并发安全

涉及的核心文件：

- `src/synapse/rd_meeting/orchestrator.py`
- `src/synapse/rd_meeting/agent_runtime.py`
- `src/synapse/rd_meeting/agent_session.py`
- `src/synapse/rd_meeting/delegation_bridge.py`
- `src/synapse/rd_meeting/work_plan.py`
- `src/synapse/rd_meeting/host_prompt.py`
- `src/synapse/rd_meeting/pipeline.py`
- `src/synapse/tools/handlers/meeting_room.py`
- `src/synapse/tools/handlers/agent.py`
- `src/synapse/agents/orchestrator.py`

---

## 2. 总体结论

当前研发会议室的整体思路是清晰的：

- 用 `rd_meeting` 承载分阶段研发流程
- 用 Host 负责主控、问卷、关卡推进
- 用 Worker 执行细分任务
- 用 work plan 约束委派行为

但在“智能体交互”层面，存在几个比较明显的问题：

1. **能力宣称与真实运行能力不一致**：系统暴露了 Agent 间发消息能力，但会议室 Worker 实际不消费消息。
2. **门禁依赖运行时状态，缺少稳定快照**：委派检查受当前节点状态影响，容易在并发/重入场景下出现误伤。
3. **主流程过于集中，逻辑分支过重**：节点执行大方法同时承载执行、重试、门控、问卷、回退等职责，维护成本高。
4. **协议依赖 prompt 文案兜底，而不是结构化约束**：核心交互靠“提醒模型重新做一遍”来修正行为，稳定性不足。
5. **角色识别机制脆弱**：通过 session id 约定来判断 Host/Worker，易受实现细节影响。
6. **工具隔离方式偏侵入式**：通过修改 agent 实例内部字段实现会议室工具裁剪，并发场景下有串场风险。

这些问题中，前两类更偏“功能正确性”，后几类更偏“系统稳定性与可维护性”。

## 3. 详细问题清单

### 3.1 `send_agent_message` 在会议室场景下可能是“死信能力”

#### 现象

研发会议室相关提示与运行时配置中，将 `send_agent_message` 视为可用协作能力的一部分：

- Host/Worker 能力文案中对 Agent 间协作有正向暗示
- 委派链路中也允许主控将任务分发给 Worker

但从实际代码链路看：

- `send_agent_message` 的实现只是将消息投递到 mailbox
- 会议室 Worker 主要通过一次性委派执行任务，然后返回结果
- 会议室执行链路里没有看到 Worker 侧稳定的 mailbox 消费闭环

这意味着：

- 消息可能被发送出去，但没有明确的消费方
- 系统对外表现为“支持智能体互发消息”，但会议室场景里并未真正闭环

#### 风险

- Host 可能基于错误能力假设做协作编排
- Prompt 会引导模型使用一个实际上无效的协作机制
- 后续定位问题时，容易把“模型不听话”误判成“模型能力问题”，而不是系统能力空转

#### 建议改造

**方案 A：先收口能力宣称（低风险，建议先做）**

- 清理会议室场景下所有暗示 Worker 可通过 `send_agent_message` 协同的提示文案
- 明确规定：会议室当前只支持“主控委派 -> Worker 执行 -> 返回结果”的单回合协作
- 若工具白名单中仍有该能力，也应在会议室场景下禁用

**方案 B：补齐真实消息闭环（中高成本）**

- 为 Worker 增加 mailbox 拉取时机
- 在 Worker 启动 prompt 中显式注入待处理消息
- 定义消息生命周期：发送、接收、确认、过期、失败处理
- 明确 Host/Worker 消息协议，避免自由文本乱传

#### 建议优先级

- **P0**

---

### 3.2 委派门禁依赖“当前节点状态”，缺少发起时快照

#### 现象

当前委派校验是围绕当前会议状态、当前节点、当前 work plan 进行的。思路本身没问题，但如果系统在委派进行时发生这些情况：

- 节点推进
- 状态刷新
- 并发触发另一个执行链路
- 计划在边界状态被锁定

那么一个原本合法的委派，可能因为“校验时拿到的是新的 current_node_id”而被拒绝。

也就是说，系统更像是在用“当前最新状态”去审查“过去已经发起的动作”。

#### 风险

- 合法委派被误判为非法
- 委派开始后 plan 被锁定，但中途异常时缺少修复路径
- 难以区分“计划真的不合法”与“状态漂移导致误伤”

#### 建议改造

**方案 A：引入委派快照（建议）**

在发起委派时固化：

- room_id
- node_id
- plan_id / plan_version
- delegator / assignee
- delegation_id

后续所有校验优先基于这份快照，而不是每次重新读取全局 current 状态。

**方案 B：为 plan 锁定增加受控修订入口**

当前一旦 `delegation_started` 后 plan 不允许再提交，过于刚性。建议提供：

- reset delegation state
- reopen plan
- attach revised plan reason

使异常恢复路径可操作、可审计。

#### 建议优先级

- **P1**

---

### 3.3 节点执行主流程过大，职责混杂

#### 现象

`rd_meeting/orchestrator.py` 中节点执行主流程承担了过多责任，包括但不限于：

- 节点启动与环境准备
- host 执行
- 自动重试
- 问卷检查
- 各类 gate 判定
- review / confirm / exception 分支
- 任务执行状态回写

一个超大方法同时处理这些职责，会带来两个问题：

1. 很难快速确认某个 gate 的真实触发路径
2. 新增一个 gate 或修改一个交互协议时，容易影响其他路径

#### 风险

- 回归风险高
- 调试门槛高
- 代码评审困难，容易“看不清是否改坏其他分支”

#### 建议改造

**方案 A：按职责拆分执行阶段（建议）**

可拆成几层：

- `prepare_node_run`
- `run_host_turn`
- `resolve_host_output`
- `apply_gate_decision`
- `finalize_node_run`

**方案 B：把各类 gate 统一成注册式策略表**

例如：

- `interactive_questionnaire`
- `result_confirm`
- `exception_gate`
- `solution_review`
- `func_solution_review`
- `task_exec_review`

统一由一个 `GateResolver` / `GateDispatcher` 处理，而不是分散在主流程各处分支判断。

#### 建议优先级

- **P2**

---

### 3.4 问卷/确认类强约束过度依赖 prompt 兜底

#### 现象

当前系统对“主控必须提交问卷”“主控不能直接结束、必须走交互工具”等要求，存在明显的 prompt 纠偏痕迹：

- 如果模型没按预期调用工具，就重新提醒一次
- 通过中文大段提示要求模型重做
- 通过多次尝试上限兜底

这说明当前系统虽然有业务规则，但没有把规则落实成强结构化执行协议。

#### 风险

- 高度依赖模型当轮理解是否稳定
- 多轮重试增加 token 成本
- 不同模型、不同 provider 下表现可能波动

#### 建议改造

**方案 A：把“必须问卷/必须确认”从 prompt 约束改成执行协议（建议）**

例如：

- 某些节点类型未提交 questionnaire 时，不允许 finish
- 某些节点类型未调用指定工具时，结果直接视为 invalid turn
- 在 runtime 层加入结构化校验，而不是“提醒模型再来一次”

**方案 B：输出统一结构化结果**

让 Host 每轮输出固定结构：

- `action_type`
- `tool_called`
- `gate_result`
- `reason`
- `next_step`

由系统判定是否满足节点约束。

#### 建议优先级

- **P2**

---

### 3.5 Host / Worker 角色识别依赖 session id 约定，机制脆弱

#### 现象

当前角色判断带有明显的“从 session id 命名规则反推角色”的特征。虽然短期能工作，但存在几个结构性问题：

- 角色语义没有成为 agent 的一等属性
- 依赖命名格式保持稳定
- session 池化/复用时，更容易出现上下文混淆
- 代码里已经出现了“必须优先看某个字段，否则会误判”的补丁式说明

#### 风险

- Host 被误判为 Worker，或反之
- 委派校验与工具权限校验走错分支
- 后续接入更多会议角色时扩展困难

#### 建议改造

**方案 A：显式角色建模（建议）**

为会议室 agent 在配置/创建时注入明确字段：

- `meeting_role = host | worker`
- `meeting_room_id`
- `meeting_profile`

运行时所有角色相关判断直接读结构化字段，不再反解 session id。

**方案 B：角色与权限绑定配置化**

把“谁能调用哪些工具、谁能提交 plan、谁能推进节点”基于 role 配置，而不是散落在多个辅助函数里做隐式判断。

#### 建议优先级

- **P0**

---

### 3.6 工具白名单通过修改 agent 实例内部状态实现，并发安全不足

#### 现象

会议室模式下，为了限制 Host/Worker 可用工具，当前实现偏向直接修改 agent 对象内部的工具集字段，然后在某些时机恢复。

这种方式的问题在于：

- agent 可能来自池化复用
- 多会议室/多节点切换时，工具状态可能串场
- 依赖“先改、后跑、再恢复”的调用顺序正确性
- 一旦中途异常，恢复可能不完整

#### 风险

- 某个会议节点使用到了不该使用的工具
- 一个场景下的裁剪结果泄漏到另一个场景
- 排查工具权限问题会非常困难

#### 建议改造

**方案 A：工具集变为 per-run 参数（建议）**

不要修改 agent 实例内状态，而是在每次执行时明确传入：

- allowed_tools
- denied_tools
- meeting_tool_profile

这样工具隔离成为“本次执行配置”，不是“实例全局状态”。

**方案 B：引入会议室工具策略对象**

例如：

- `MeetingToolPolicy.for_host(node_type)`
- `MeetingToolPolicy.for_worker(task_type)`

由策略对象统一产出本轮可用工具集。

#### 建议优先级

- **P1**

## 4. 优先级建议

建议按以下顺序逐项核对：

### 第一批：先验证功能正确性

1. **P0 - `send_agent_message` 是否在会议室场景下真实可用**
2. **P0 - Host / Worker 角色判定是否存在误判路径**

目标：先确认系统有没有“表面支持、实际失效”的核心交互能力。

### 第二批：再处理状态一致性与隔离性

3. **P1 - 委派门禁是否应该切换到快照校验**
4. **P1 - 工具白名单是否应该改为 per-run 配置**

目标：降低并发、重入、复用场景下的隐性 bug。

### 第三批：最后做结构治理

5. **P2 - 节点执行主流程拆分**
6. **P2 - 问卷/确认协议结构化**

目标：提升长期可维护性与模型行为稳定性。

---

## 5. 推荐核对方式

后续逐项核对时，建议每一项都按下面的模板推进：

### 核对模板

#### 1）问题是否真实存在

- 代码入口在哪里
- 调用链路是什么
- 是否有反例或兜底逻辑
- 是否只是在某些模式下才会触发

#### 2）影响范围多大

- 只影响 rd_meeting，还是会波及全局 agent orchestration
- 只影响 Worker，还是 Host 也受影响
- 是功能错，还是只是维护成本高

#### 3）最小改造方案是什么

- 先止血
- 再优化
- 是否可以分阶段上线

#### 4）如何验证

- 单测补哪里
- 集成测试补哪里
- 是否需要构造 Host/Worker/plan/node 的最小复现用例

---

## 6. 建议下一步

建议下一轮先从以下两项开始逐条核对：

1. **`send_agent_message` 在研发会议室里是否真的没有消费闭环**
2. **Host / Worker 角色识别是否存在误判与补丁式修复**

这两项都更偏“基础交互正确性”，适合先确认。

如果确认无误，再继续进入：

- 委派快照化
- 工具隔离改造
- 主流程拆分
- 问卷协议化
