# 模块功能技能

以需求澄清文档为分析起点与唯一需求依据，在 **FUNCTIONAL_ARCH 功能模块粒度**识别需改造的模块清单，明确各模块**要做到什么、满足哪些需求、如何验证**，再结合产品架构文档与代码确认核心类。**严禁臆断想象**，所有结论必须有实际存在的内容作为依据。

> **粒度原则**：本技能只到**功能架构模块级**，不在此阶段拆细到页面、接口、按钮、函数等功能点；细粒度拆解由下游「函数级方案」技能完成。若需求澄清文档中的功能要点更细，须**向上归并**到对应架构模块，禁止向下拆细。

## 研发会议室：工单目录读代码 / 文档

`room_opened` 已将产品资产落盘到工单目录。**只读下列路径，禁止臆造目录。**

| 用途 | 路径约定 | 说明 |
|------|----------|------|
| 工单根目录 | `{WORK_ORDER_DIR}` | 系统提示「产品工作区路径」或委派参数，形如 `work/<scope_id>/` |
| 需求澄清文档 | **固定路径（勿传参）** | `{WORK_ORDER_DIR}/archive/需求分析/req_clarify/需求澄清.md` |
| 产品代码 | `{PRODUCT_CODE_ROOT}/<repo_name>/` | 默认 `{WORK_ORDER_DIR}/code/<repo_name>/` |
| 产品文档 | `{PRODUCT_DOC_ROOT}/<doc_type>/` | 默认 `{WORK_ORDER_DIR}/doc/<doc_type>/`；架构文档在 `产品架构/` |

- **需求澄清（最高优先级）**：本技能的一切需求理解、模块识别、范围界定**必须**基于固定路径 `{WORK_ORDER_DIR}/archive/需求分析/req_clarify/需求澄清.md`；未读取该文档前不得开展后续分析。**仅此唯一路径**，禁止尝试其他路径或回退路径。
- **FUNCTIONAL_ARCH / TECH_ARCH**：优先读 `{PRODUCT_DOC_ROOT}/产品架构/`
- **代码确认**（Step 3）：在 `{PRODUCT_CODE_ROOT}` 各仓库目录内检索，路径标注为 `<repo_name>/相对路径`

## Parameters

| Parameter | 必填 | 说明 / 示例 |
|-----------|------|----------------|
| `PROD` | 是 | 产品名称 |
| `WORK_ORDER_DIR` | 是 | 工单工作目录（研发会议室注入），如 `work/21881451`。**必须**提供，用于解析固定需求澄清路径 `{WORK_ORDER_DIR}/archive/需求分析/req_clarify/需求澄清.md` |
| `PRODUCT_CODE_ROOT` | 否 | 产品代码根目录，默认 `{WORK_ORDER_DIR}/code` |
| `PRODUCT_DOC_ROOT` | 否 | 产品文档根目录，默认 `{WORK_ORDER_DIR}/doc` |

> **知识来源（优先级从高到低）**：
> 1. **需求澄清文档**（固定路径见上表）— 需求范围、功能要点、澄清结论、约束与依赖的**唯一权威来源**；模块目标与需求映射必须可回溯至该文档中的具体条目
> 2. **产品架构文档**（FUNCTIONAL_ARCH、TECH_ARCH）— 模块列表与架构划分的**粒度基准**
> 3. **代码** — 验证模块存在性与核心类的最终事实依据
>
> 任何无法从上述来源验证的结论必须标注 `[待确认]`，严禁臆断想象。

## 核心约束（违反本技能视为未完成）

### A. 技能执行输出限制

**整个技能执行过程中，大模型的输出应遵循以下原则：**

1. **生成文档阶段**：按照下方「输出内容格式」章节的结构输出结果
2. **异常中止时**：可直接输出中止原因和缺失信息，无需走文档格式

### B. 需求澄清文档优先（强制约束）

- **必须先读需求澄清文档**：Step 1 成功读取 `{WORK_ORDER_DIR}/archive/需求分析/req_clarify/需求澄清.md` 之前，禁止进行模块识别、范围界定或代码确认。
- **分析必须基于需求澄清文档**：模块改造范围、IN/OUT 界定、模块目标与「满足的需求」映射、新增模块判定，均须引用需求澄清文档中的具体章节或条目（范围定义、功能要点、澄清结论、约束与依赖）。
- **禁止绕过需求澄清文档**：不得直接依据原始需求描述、口头理解或 FUNCTIONAL_ARCH 推断需求范围；原始需求仅作背景展示，**不得**作为模块识别的依据。
- 需求澄清文档与 FUNCTIONAL_ARCH 描述冲突时，以**需求澄清文档**为准；冲突点须在输出中标注说明。

### C. 模块粒度约束（强制约束）

- **划分单位**：输出清单的每一行对应 **FUNCTIONAL_ARCH「核心功能详解」中的一个功能模块**，模块名称必须与 `FUNCTIONAL_ARCH.md` 保持一致。
- **禁止细拆功能点**：不得输出页面级、接口级、字段级、函数级等功能点清单；不得单独设立「需求功能拆分」章节罗列细粒度功能点。
- **向上归并**：需求澄清文档中的多条功能要点若归属同一架构模块，须合并为**一条模块记录**，在「满足的需求」列汇总对应澄清条目。
- **向下留白**：模块内具体改造点、函数清单、伪代码由下游「函数级方案」技能负责，本技能**不涉及**具体改造方案或实现细节。

### D. 模块三要素（强制约束）

每个需改造模块（含修改与新增）**必须**写清以下三项，缺一不可：

| 要素 | 要求 |
|------|------|
| **模块目标** | 本需求下该模块**要做到什么**（能力/行为变化，1～3 句，模块级表述） |
| **满足的需求** | 对应需求澄清文档中的哪些条目（范围定义、功能要点、澄清结论）；须可回溯，不得超出 IN 或包含 OUT |
| **验证方式** | **如何验证**模块目标已达成：可执行的验收条件 + 验证手段（如联调场景、接口调用、页面操作、单元/集成测试、日志或配置检查等） |

验证方式须具体可执行，禁止仅写「测试通过」「功能正常」等空泛描述。

### E. 产品知识体系事实依赖（强制约束）

- 模块识别**必须严格依赖**以下三类事实来源，且按优先级使用，不得凭空臆断或想象：
  | 优先级 | 事实来源 | 说明 |
  |--------|---------|------|
  | **1（需求依据）** | **需求澄清文档** | `{WORK_ORDER_DIR}/archive/需求分析/req_clarify/需求澄清.md` — 范围定义、功能要点、澄清结论、约束与依赖 |
  | **2（架构参考）** | **产品架构文档** | `{PRODUCT_DOC_ROOT}/产品架构/`（FUNCTIONAL_ARCH、TECH_ARCH） |
  | **3（代码验证）** | **代码** | 优先 `{PRODUCT_CODE_ROOT}/<repo_name>/`；通过文件系统或检索工具读取，禁止臆造路径 |
- 模块识别中凡涉及**文件路径、接口、类名**等结论，必须在代码或文档中有对应支撑依据，并标注来源。
- 无法通过上述事实来源验证的条目必须标注 **`[待确认]`**，不得虚构。
- 若某文档源未获取成功，涉及该文档的分析标注 `[待补充-文档未获取]`。

### F. 输出范围约束

- 本技能**只输出需改造的模块清单**（含模块目标、满足的需求、验证方式、核心类），不涉及具体改造方案、改造内容或实现细节
- 超出功能架构模块列表的改造范围，标识为**新增模块**

### G. 事实驱动的模块确认（强制约束）

- `FUNCTIONAL_ARCH.md` 是**指导性文档，不一定完全正确**，必须通过实际代码确认其准确性
- 须先从需求澄清文档理解 IN 范围，再按 ARCH 模块列表判断哪些模块涉及改造，并通过代码确认
- 代码确认可能发现功能架构描述与实际代码不一致的情况，此时以代码为准
- 每个修改模块都必须经过代码确认
- 每个新增模块需确认功能架构中确实没有可覆盖的模块
- 所有确认结论必须有具体的文件路径或文档引用作为依据

## 工作流程

```
Step 0 — 参数校验与环境准备
  0a. 校验必填参数：PROD、`WORK_ORDER_DIR`（缺任一则**中止**）
  0b. 解析固定路径（**唯一地址，不可变更、不可回退**）：
      `CLARIFY_DOC_PATH = {WORK_ORDER_DIR}/archive/需求分析/req_clarify/需求澄清.md`
      （勿由调用方传入 `CLARIFY_DOC` 或其他路径参数）
  0c. **解析工单目录**：`CODE_ROOT = PRODUCT_CODE_ROOT`（缺省 `{WORK_ORDER_DIR}/code`），`DOC_ROOT = PRODUCT_DOC_ROOT`（缺省 `{WORK_ORDER_DIR}/doc`）

Step 1 — 获取项目资料（需求澄清文档优先）
  1a. **仅读取** `CLARIFY_DOC_PATH`（即 `{WORK_ORDER_DIR}/archive/需求分析/req_clarify/需求澄清.md`）；若不存在则**立即中止**，注明该固定路径，**不得**尝试其他路径
  1b. 在 `{DOC_ROOT}/产品架构/` 读取 FUNCTIONAL_ARCH、TECH_ARCH 相关 `.md`；FUNCTIONAL_ARCH 缺失则**中止**

Step 2 — 基于需求澄清与功能架构识别涉及模块
  2a. **基于需求澄清文档**（`CLARIFY_DOC_PATH`）提取以下结构化信息（本步骤的核心输入，不可跳过）：
      - **范围定义**：IN（涉及范围）和 OUT（排除范围）
      - **功能要点**：需求涉及的能力描述（作为模块「满足的需求」的素材，**不**逐条输出为功能点清单）
      - **澄清结论**：关键确认点的总结
      - **约束与依赖**：模块依赖、技术约束、数据依赖
  2b. 读取 FUNCTIONAL_ARCH.md，提取"核心功能详解"章节中的**功能模块列表**（每个模块的名称、功能描述、涉及的源代码文件），记为 ARCH_MODULE_LIST
  2c. **按 ARCH 模块粒度**确定候选改造模块 CANDIDATE_MODULES（不得细于 ARCH_MODULE_LIST）：
      - 将 2a 中与 IN 相关的内容**映射**到 ARCH_MODULE_LIST 中最匹配的模块（一条或多条澄清条目可映射到同一模块）
      - OUT 范围内的内容不得纳入任何模块的「满足的需求」
      - 若某澄清要点无法映射到任何 ARCH 模块 → 进入 Step 3d 新增模块流程
      - **禁止**在此步骤生成独立于 ARCH 模块的细粒度功能点列表

Step 3 — 模块确认、三要素编写与代码确认
  对 CANDIDATE_MODULES 中的每个模块，依次执行：

  3a. **编写模块三要素**（须引用需求澄清文档具体条目）：
      - **模块目标**：本需求下该模块应达成的能力/行为（模块级，1～3 句）
      - **满足的需求**：汇总映射到该模块的需求澄清条目（章节/要点编号或原文摘要）
      - **验证方式**：可执行的验收条件与验证手段（场景 + 预期结果 + 工具/路径）

  3b. **代码确认**：
      功能架构是指导性文档，必须通过实际代码确认该模块是否需要改造及核心类是否准确：
      - 在 `{PRODUCT_CODE_ROOT}`（或 `{WORK_ORDER_DIR}/code`）下各 `<repo_name>/` 目录中搜索模块关键类名/接口名
      - 在关键模块路径下搜索相关业务逻辑代码
      - 读取模块核心代码文件（路径格式：`<repo_name>/相对路径`）
      - **识别核心类**：从搜索结果中提取该模块的核心类（Controller/Service/DAO/Model 等），记录类名与**工单代码目录内**文件路径

  3c. **确认结论**（以代码为准）：
      - 代码确认该模块确实存在且需要改造 → 确定为**修改模块**，记录核心类列表和来源路径
      - 代码发现功能架构描述与实际代码不一致 → 以代码为准，调整核心类与确认依据
      - 代码确认该模块已能覆盖需求、无需改造 → 从改造清单中排除
      - 代码无法验证 → 标注 `[待确认]`

  3d. **新增模块判定**（对 2c 中无法映射到 ARCH 的需求范围）：
      - 若需求范围无法由 ARCH_MODULE_LIST 中任何已有模块承载
      - 或已有模块经代码确认后，功能范围差异过大、不适合在该模块内改造
      - 则标识为**新增模块**，须同样填写模块三要素 + 新增原因，并通过代码检索识别可能的核心类/落点

Step 4 — 生成模块功能文档
  4a. 汇总：需改造的模块清单（修改模块 + 新增模块），每条含模块目标、满足的需求、验证方式
  4b. 按照「输出内容格式」章节的结构输出结果
```

## Error Handling

| 情况 | 处理 |
|------|------|
| 缺少必填参数（PROD 或 `WORK_ORDER_DIR`） | **中止**，列出缺失项 |
| 需求澄清文档 `{WORK_ORDER_DIR}/archive/需求分析/req_clarify/需求澄清.md` 不存在 | **立即中止**（本技能无法在无需求澄清文档的情况下执行），注明该固定路径 |
| 需求澄清文档存在但无法提取 IN 范围或功能要点 | **中止**，说明文档缺失的关键章节 |
| `{DOC_ROOT}/产品架构/` 无 FUNCTIONAL_ARCH | **中止** |
| TECH_ARCH 缺失 | 继续，仅以 FUNCTIONAL_ARCH 为准 |
| 代码检索无结果 | 标注 `[待确认]`，不得虚构内容 |
| 某模块缺少模块目标、满足的需求或验证方式任一字段 | **不得输出该模块**，补全或标注 `[待确认]` 后输出 |
