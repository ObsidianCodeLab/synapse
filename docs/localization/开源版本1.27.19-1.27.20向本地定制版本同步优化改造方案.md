## 2. 功能详解与本地差距

### 2.1 `main.rs` — 流式 UTF-8 边界修复（P0，`ee327ab4`，Issue #602）

**问题**：Tauri 壳 `backend_fetch` SSE 与 `pip_install` 管道按 chunk 独立 `String::from_utf8_lossy`，多字节字符（中文/emoji）跨 chunk 边界时被拆成两个 U+FFFD，聊天出现「黑色菱形乱码」。

**上游改法**：新增 `take_valid_utf8_prefix`，在 `backend_fetch` 与 pip stdout/stderr 读取处累积 pending buffer，只解码完整 UTF-8 前缀。

**本地现状**：**无** `take_valid_utf8_prefix`；本地 main.rs 在 v1.27.19 已合看门狗/rd_terminal 等定制块。

**适配**：

- 块级移植 helper + 两处读流改造；
- **禁止**整文件覆盖 main.rs；
- 与 v1.27.19 已合的 `log_to_file`→`record_event`、看门狗断路器共存。

**验收**：Tauri 聊天流式中文无 U+FFFD；引导 pip 安装中文输出正常。

### 2.2 Issue #615 — Message Parts 有序投影（P0，`d954d8d6`）

**问题**：实时 SSE 与持久化历史是两套管线；重载/多窗口切换后 plan 卡片、已答 ask_user、附件、中断流 reconciliation 丢失或错乱。

**上游改法**：

| 前端 | 新 `messageParts.ts`、`MessageParts.tsx`、`AskUserSummary.tsx`、`attachments.ts`；`FlatMessageItem` / `MessageBubble` 统一走 parts 渲染 |

**本地现状**：上述文件 **均不存在**；`ChatView.tsx` / `chatHelpers.ts` / `types.ts` 仍为旧模型。

**适配**：

- 新增文件可直接品牌化引入（路径 `openakita`→`synapse` 仅后端 import）；
- `ChatView.tsx`、`MessageBubble.tsx` 等 **块级 merge**，保留本地 tab/研发相关 hook 若有交叉；

**验收**：reload / 多窗口后 plan 卡片、已答 ask_user、图片附件仍正确；中断流 reconciliation 不丢 authoritative 答案。

### 2.10 `server.py` — 内置用户手册部署优化（P2，**评审结论：合入**）

**功能说明**（非研发云/语雀，与定制路由无关）：

- Tauri 侧边栏「文档」→ iframe 加载 `{apiBase}/user-docs/`（`App.tsx` `view === "docs"`）。
- 启动 `create_app()` 时将包内 `docs_dist/` 部署到工作区 `data/docs/v{version}/`，FastAPI 挂载 `/user-docs/`。

**v1.27.19 本地行为**：每次 `create_app()` **整目录删除再全量复制**（`rmtree` + `copytree`）。

**v1.27.20 上游改法**（`0867f3f1`）：

| 旧 | 新 |
|----|-----|
| 每次全量删目录再复制 | `_docs_version_matches_bundled` 比对 bundled 与已部署目录 |
| Windows 上 StaticFiles 仍挂载时易失败 | 一致则 **跳过**；有变更则 `_sync_docs_tree` **增量同步** |
| 集成测试反复 `create_app` 重复部署 | 避免重复替换同一版本目录 |

**对 Synapse 定制的影响**：

- **无** yuque/gitnexus/dev_iwhalecloud 交叉；只改 `_deploy_docs` 及新增 helper。
- **对 Tauri 有益**：常用内置文档页时启动更快、Windows 更稳。
- **合入方式**：块级移植 `_docs_version_matches_bundled` / `_sync_docs_tree` / `_record_docs_version`，重写 `_deploy_docs` 调用逻辑。

**评审决策**：**合入**（P2 优先于 org 超时）。

### 2.11 `orgs/runtime.py` — 移除节点 `max_task_seconds`（P2，**评审结论：跟上游合**）

**机制**：组织多智能体节点 `runtime_overrides.max_task_seconds` 对 `agent.chat` 做 `asyncio.wait_for` 硬超时；超时返回软失败并标记任务 CANCELLED。

**上游 v1.27.20（`9d511347`）**：删除 per-node 硬超时，改由 org **`_command_watchdog`**（命令级卡住）处理；`check_permission` patch 改为 `getattr` 防御。

**本地影响分析**：

| 检索范围 | 结果 |
|----------|------|
| 前端 OrgEditor 等 | **无** `max_task_seconds` 配置 UI |
| 仓库 org yaml/json 样例 | **无** 实际配置 |
| 研发云/产品工作台/GitNexus | **无** 依赖 |
| `runtime.py` / `models.py` | 仍有实现与注释 |

**影响面**：

- 未手工配置 `max_task_seconds` 的组织：**几乎无感**。
- 曾在 org JSON 手工配置该字段：**合入后失效**，节点 hang 改由 command watchdog 处理（粒度为整条用户命令，非单次 delegate）。

**评审决策**：**跟上游合入**（定制模块不依赖；前端未暴露该能力，实际使用概率低）。同时合入 `getattr` permission patch。
