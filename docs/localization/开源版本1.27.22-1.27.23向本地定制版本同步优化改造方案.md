## 2. 功能详解与本地差距

### 2.1 LLM 端点批量删除（P0，#644）

**上游动机**：用户通过导入/批量添加产生大量端点后，需一次性删除多行；单删会多次 reload，慢且易中间态不一致。

**上游改法（块级对照，禁止整文件拷贝）**：

#### 2.1.1 后端 `config.py`

在 `DeleteEndpointRequest` 之后新增：

```python
class DeleteEndpointsRequest(BaseModel):
    names: list[str]
    endpoint_type: str = "endpoints"
    clean_env: bool = True
```

在 `save_endpoints` 与 `delete_endpoint_by_name` 之间新增路由：

- `DELETE /api/config/endpoints`
- 调用 `mgr.delete_endpoints(body.names, ...)`
- 返回 `removed` / `removed_count` / `not_found` / 单次 `_trigger_reload`

**本地路径**：`src/synapse/api/routes/config.py`（约 940 行 `delete_endpoint_by_name` 之前插入）。

**注意**：本地 `config.py` 含研发云/工作台无关扩展时 **只插入新 class + 新 route**，勿覆盖现有 save/validate hunk。

#### 2.1.2 后端 `endpoint_manager.py`

在 `delete_endpoint()` 之后新增 `delete_endpoints()`：

- 一次 `_read_json_versioned` + 过滤 `wanted` 名称集合
- 批量移除；`clean_env` 时对 **removed** 集合里的 `api_key_env` 做「仍被剩余端点引用则保留」逻辑（与单删一致，但对多个 env_var 去重处理）
- 单次 `_write_json`

**本地路径**：`src/synapse/llm/endpoint_manager.py`（约 363 行 `delete_endpoint` 结束处插入 ~46 行）。

#### 2.1.3 前端 `LLMView.tsx`（**高风险文件，必须块级**）

上游新增能力摘要（按 hunk 合入，勿整文件）：

| 块 | 内容 |
|----|------|
| state | `selectedEndpointNames: Record<EndpointType, Set<string>>` |
| helpers | `selectedNamesForType` / `setSelectedNamesForType` / `toggleEndpointSelected` / `setAllEndpointsSelected` / `allEndpointsSelected` |
| effect | 端点列表变化时 prune 无效选中项 |
| action | `doDeleteSelectedEndpoints(names, endpointType)` → `DELETE .../api/config/endpoints` |
| UI | `renderBatchDeleteToolbar`；三张表（chat / compiler / stt）各加 Checkbox 列 + `colSpan` 调整 + 选中行 `bg-primary/5` |

**本地已有、合入时必须保留**：

- `EndpointType` 含 `compiler_endpoints` / `stt_endpoints`
- `originalName` / `editingOriginalName` 编辑与 save 参数
- 本地告警文案（无可用聊天端点等）
- 已有 `Checkbox`、`cn` import（无需从 upstream 覆盖 import 段）

**合入顺序建议**：

1. 先合 Python + 单测，跑 pytest 通过
2. 再按 upstream diff 在 **chat 表** 插入 checkbox + toolbar
3. **复制同一模式** 到 `savedCompilerEndpoints` / `savedSttEndpoints` 两表（upstream 已含三表；对照 `git show de7e34ba -- LLMView.tsx` 中 compiler/stt 段）
4. 批量删除文案可保留 upstream 中文或迁入 i18n（非阻塞）

**与会议室/工作台关系**：**无**。LLM 配置页不涉及 `rd_meeting` / `workbench_*` / delegation。

### 2.2 插件 UI CDN 版本锁定（P1，#652）

**上游问题**：unpkg 上 `@18` 与无版本 `@babel/standalone` 会静默升级，导致 JSX eval bootstrap 与 Babel 7.29+ 不兼容。

**锁定版本**（三行替换，每个 `index.html` 相同模式）：

```html
<script crossorigin src="https://unpkg.com/react@18.3.1/umd/react.production.min.js"></script>
<script crossorigin src="https://unpkg.com/react-dom@18.3.1/umd/react-dom.production.min.js"></script>
<script src="https://unpkg.com/@babel/standalone@7.29.7/babel.min.js"></script>
```

**上游修改的 13 个插件**（本地需同步 CDN 行）：

|------|-------------|------|
| avatar-studio | 否 | 改 3 行 |
| clip-sense | 否 | 改 3 行 |
| ecommerce-image | 否 | 改 3 行 |
| excel-maker | 否 | 改 3 行 |
| fin-pulse | 否 | 改 3 行 |
| footage-gate | 否 | 改 3 行 |
| manga-studio | 否 | 改 3 行 |
| media-post | 否 | 改 3 行 |

```powershell
Get-ChildItem D:\github\openakita_jyhk\plugins -Directory | ForEach-Object {
  $f = Join-Path $_.FullName "ui\dist\index.html"
```

### 2.3 SignPath Windows 发布签名（P3，#653，默认跳过）

**上游能力**：在 `desktop_release` job 的 `Collect installer bundles` 之后：

1. 输出 `windows_installer` / `signpath_output_dir`
2. upload unsigned exe artifact
3. `signpath/github-action-submit-signing-request@v2`（`project-slug: openakita` 等）
4. 用签名产物替换原 exe 并重跑 Tauri updater 签名

**本地差距**：

- `.github/workflows/release.yml` 已有 Synapse 发版流程，**无** SignPath 块
- upstream slug / 证书名均为 **openakita**，直接合入会违反品牌化且 SignPath 租户可能无对应项目

**策略**：

| 方案 | 条件 | 做法 |
|------|------|------|
| **A（推荐）** | 暂无 SignPath Synapse 租户 | **跳过** #653 |
| **B** | 已在 SignPath 配置 `synapse` project + policy + cert | 块级合入 `collect_installers` 扩展 + 4 个 step；**全部 slug 改为 synapse**；secrets 增加 `SIGNPATH_API_TOKEN` |

**禁止**：整文件 `checkout upstream release.yml`（会丢失本地 Synapse 发版差异）。
