## 2. 功能详解与本地差距

### 2.1 sessions.json 并发写串行化（P0，#665）

**上游问题**：`persist()`、debounce、线程池可能并发写 `sessions.json`，与 v1.27.22 原子写叠加仍可能竞态；序列化时读到正在被 append/trim 的 live 结构。

**上游改法**：

- `sessions/manager.py`：manager 级可重入锁保护快照 + 原子写；compact JSON 减小临界区。
- `sessions/session.py`：`to_dict()` / 快照返回 locked deep-copy。
- `utils/atomic_io.py`：小改动（与 compact 配合）。

**本地适配**：

- **块级**合入 lock + snapshot hunk；**保留** v1.27.22 已合的 `atomic_json_write` 与 **PR-N1** 会话 ID 校验、invalid_id 采样日志。
- 单测：`tests/unit/test_session.py` 新增并发用例 **追加合入**。

**与会议室关系**：会话持久化为会议室/委派共用基础设施；合入**有利于** rd_meeting 稳定性，但须回归 `pool_session_id` 会话是否正常落盘。

---

### 2.2 serve 启动顺序与 API late-bind（P0，#664）

**上游改法**：

- `main.py`：`start_api_server(agent=None, session_manager=None, …)` 先起 HTTP；Agent.initialize + init_core_services 后再 bind 到 `app.state`。
- `api/server.py`：late-bind agent/orchestrator/gateway/pending plugin routes；`Agent.initialize()` 串行防重入。
- `core/agent.py`：少量 initialize 守卫。

**本地差距**：

- `server.py` 挂载 **inbox / yuque / gitnexus / dev_iwhalecloud** 及 `_start_inbox_service` 生命周期。
- 本地 `main.py` 可能有 Synapse 定制启动日志/路径。

**本地适配**：

- **禁止** `git checkout upstream -- server.py main.py`。
- 按 upstream diff **分段**移植：先 API 早启动 + late-bind 框架，再确认 inbox 启动/停止钩子仍在正确 phase。
- 单测：`tests/unit/test_serve_startup_order.py` **新增合入**。

**风险**：若 late-bind 顺序错误，Setup Center 可能在 Agent 未就绪时调用 chat/会议室 API → 合入后必测 `/api/health` 早就绪 + 首轮 chat。
