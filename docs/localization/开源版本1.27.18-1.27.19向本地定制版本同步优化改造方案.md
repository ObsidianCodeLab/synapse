## 2. 功能详解与本地差距

### 2.1 `bug_report.py` — WER 权限防护（**不合入**）

**上游问题**：Web 路径 `_build_bug_zip` → `_add_windows_crash_artifacts` 扫描 `%ProgramData%` WER 时，非管理员可能 `PermissionError` 导致整包失败。

**本地结论**：

- Tauri 反馈走 SynapseService + `export_diagnostic_bundle`，**不调用** `_build_bug_zip`，也不扫描 WER ReportArchive。
- Web `/api/bug-report` **不在维护范围**。
- **无需**移植 `92b1b617`。

若日后恢复 Web 反馈，再单独 cherry-pick 该 commit。

---

### 2.2 `main.rs` — 看门狗滑动窗口断路器（P0）

**问题**：v1.27.18 本地为 **30s 单点冷却**（`WATCHDOG_RELAUNCH_COOLDOWN_SECS`）。确定性崩溃循环（如每分钟崩一次）会 **无限拉起**。

**上游改法**（`d57df077`）：

| 旧 (1.27.18) | 新 (1.27.19) |
|--------------|--------------|
| marker 存单个 timestamp | marker 存 **多行** epoch 秒 |
| 距上次 < 30s 跳过 | **180s 窗口内 ≥ 3 次** 断路停手 |
| | 窗口滑动过期后自动复位 |

常量：

- `WATCHDOG_BREAKER_WINDOW_SECS = 180`
- `WATCHDOG_BREAKER_MAX_RELAUNCHES = 3`

**本地现状**：`main.rs` 仍为 `WATCHDOG_RELAUNCH_COOLDOWN_SECS` + 单 timestamp 逻辑。

**适配**：仅替换看门狗 marker 注释、常量、`run_watchdog()` 断路段落；保留 rd_terminal、服务清理、`prepare_relaunch`、self-heal 等本地块。

**验收**：3 分钟内连续崩溃 3 次后看门狗停手；偶发单次崩溃仍 `--auto-restarted` 恢复。

---

### 2.2b `main.rs` — `log_to_file` → `record_event`（P0，v1.27.18 漏合）

**背景**：上游 v1.27.18 已在 `log_to_file` 末尾调用 `crash_handler::record_event(msg)`，使看门狗/自愈/健康探测日志进入 crash ring buffer，随 `crash.log` 与 minidump 一并出现在 `export_diagnostic_bundle` 的 `global_logs/crash.log` 上下文里。

**本地现状**：`log_to_file` **未**转发 `record_event`；`a62c85d9` 的 UTF-8 修复在无转发时几乎不会被触发，但合入 UTF-8  fix 时应 **一并补上** 一行转发（不必搬入上游 `rotate_autostart_log_if_needed` 整段，除非 autostart.log 体积已成问题）。

```rust
// log_to_file() 末尾追加：
crash_handler::record_event(msg);
```

---

### 2.3 `crash_handler.rs` — UTF-8 安全截断（P0）

**问题**：`record_event` 在字节 512 处硬切；CJK 日志可能在多字节字符中间 panic，且在 UI/IPC/tray 回调路径上可能升级为进程 abort。

**上游修复**（`a62c85d9`）：`is_char_boundary` 回退切点。

**本地现状**：仍为 `&msg[..EVENT_MAX_LEN]` 危险写法。

**适配**：块级移植 `record_event` 截断逻辑；保持 `com.synapse.setupcenter` / `~/.synapse/crashdumps` 品牌化路径不变。

**验收**：长中文 autostart 诊断行写入时不 panic；Tauri 反馈打包的 `global_logs/crash.log` 正常。

---

### 2.4 移除 global-shortcut / notification（P0）

**上游动机**（`39bdff13`）：`tauri-plugin-global-shortcut` 被怀疑与 WebView2/tao 事件循环重入崩溃相关；`notification` 无调用方。

**删除范围**：

- `Cargo.toml` / `package.json` / `package-lock.json`
- `capabilities/default.json` 相关权限
- `main.rs` `.plugin(tauri_plugin_global_shortcut|notification)`
- `platform/index.ts` 的 `registerGlobalShortcut` / `sendNotification`
- `App.tsx` 全局快捷键 `useEffect`

**本地差异**：

| 项 | 上游 | 本地 Synapse |
|----|------|--------------|
| 快捷键 | `Ctrl+Shift+A` | **`Ctrl+Shift+C`** |
| `sendNotification` | 已删 | 仅 platform 定义，**无业务调用** |
| 插件 | 已删 | **仍在** |

**推荐（Tauri 稳定性优先）**：

1. 与上游对齐 **移除插件**
2. 删除 `App.tsx` 中 `Ctrl+Shift+C` 整块及 `registerGlobalShortcut` import
3. **保留** `platform/index.ts` 的 `prepare_relaunch`（删插件段时勿误删）
4. 唤窗能力：托盘 / 单实例聚焦（已有）；后续如需全局热键再评估非 global-shortcut 方案

**用户可见变化**：失去全局 `Ctrl+Shift+C`；托盘仍可用。
